# Cottonwood Harbor Product Commandments

Internal Use Only

This is the internal constitution for Cottonwood Harbor product decisions.

Customers do not need to read these commandments. They should feel them every time they use Harbor Vault.

Every product decision, RFC, feature, pull request, and release should be measured against this document.

## 1. Organization First

Every feature begins with the Organization as the primary unit of ownership, security, collaboration, and context.

## 2. Simplicity First

Power without complexity.

The simplest solution that solves the problem wins.

## 3. AI Is Native

AI is a teammate, not a feature.

If AI can reduce friction, it should.

## 4. One Platform

Never build disconnected products.

Every feature should strengthen Harbor Vault as a unified operating system.

## 5. Opinionated Defaults

Configuration is a last resort.

The platform should make smart decisions before asking the user to make them.

## 6. Progressive Disclosure

Users should only see what they need today.

Advanced capabilities should reveal themselves naturally as Organizations grow.

## 7. Every Screen Has One Job

A user should immediately understand why they are on a page and what they can accomplish there.

If a screen has multiple primary purposes, reconsider the design.

## 8. Predictable Over Flexible

Users should know what Harbor Vault will do.

Avoid hidden behavior, surprising automation, and inconsistent interfaces.

## 9. Respect The User's Time

Every click, prompt, dialog, and configuration step should justify its existence.

If a workflow can be shortened without sacrificing clarity, shorten it.

## 10. Explainable Intelligence

When Harbor AI makes a recommendation or performs an action, users should understand what happened and why.

Trust is earned through transparency.

## 11. Consistency Creates Confidence

The same action should work the same way throughout the product.

Shared patterns reduce learning and increase trust.

## 12. Build For The Next Ten Years

Avoid shortcuts that create long-term product debt.

Favor maintainable architecture, clear models, and thoughtful evolution over rapid feature accumulation.

## The Final Question

Before approving any feature, RFC, or pull request, ask:

Does this make Harbor Vault easier to use, easier to understand, or more valuable for the Organizations we serve?

If the answer is not clearly yes, revisit the design.
