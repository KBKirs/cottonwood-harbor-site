# Cottonwood Harbor RFC Process

Request for Comments documents create a stable target before significant Harbor Vault features are built.

Every significant product decision should have an RFC before implementation.

## Why RFCs Exist

RFCs help Cottonwood Harbor:

- Clarify the problem before building
- Capture why a decision was made
- Give Codex a stable implementation target
- Preserve product history
- Reduce rework
- Keep Harbor Vault aligned with the Brand and Product Bible

## RFC Ownership

CEO:

- Sets vision
- Provides customer feedback
- Makes final decisions
- Prioritizes what moves forward

Chief Product and Design Officer:

- Owns product architecture
- Owns UX and information architecture
- Defines Harbor AI behavior
- Writes and maintains RFCs
- Reviews Codex implementation against RFCs

Codex Lead Engineer:

- Reviews RFCs for implementation implications
- Identifies technical risks
- Implements accepted RFCs
- Tests and deploys approved work

## RFC Statuses

- Draft
- In Review
- Accepted
- In Build
- Shipped
- Superseded

## Required RFC Sections

Every RFC should include:

- Problem statement
- Goals
- Non goals
- User stories
- UI direction or mockup notes
- Workflow
- Permissions
- Data model implications
- Harbor AI behavior
- Future considerations
- Acceptance criteria
